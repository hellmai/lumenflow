Use this flow when a repository already has LumenFlow installed and you want the latest
`wu:brief` defaults, onboarding docs, and vendor assets.

## Existing Installs

<Steps>

1. **Upgrade the installed packages**

   ```bash
   pnpm lumenflow:upgrade --latest
   ```

2. **Verify the upgrade**

   ```bash
   pnpm gates --docs-only
   ```

</Steps>

## What Changes Automatically

`lumenflow:upgrade` handles everything in a single atomic transaction:

- Runtime `wu:brief` defaults update immediately.
- **LUMENFLOW.md** and `.lumenflow/constraints.md` are force-synced from the installed CLI templates (these files are fully LumenFlow-managed).
- **AGENTS.md** and vendor bootstrap files (CLAUDE.md, `.cursor/rules/lumenflow.md`, `.windsurf/rules/lumenflow.md`) are updated via merge-block markers — user content outside the `LUMENFLOW:START`/`END` markers is preserved.
- Onboarding docs and vendor assets (skills, hooks) are refreshed.
- Project-local `.lumenflow/templates/` files are optional. Only create or edit them when you
  want to override the built-in prompt composition.
- Lane lifecycle commands now read lane definitions from `workspace.yaml >
software_delivery.lanes.definitions` only.

<Aside type="tip" title="No separate docs:sync step needed">
  Prior to v3.16, upgrade instructions advised running `pnpm docs:sync --force` as a separate
  post-upgrade step. This is no longer needed — `lumenflow:upgrade` runs doc sync internally.
</Aside>

<Aside type="note" title="Lane source of truth">
  `.lumenflow.lane-inference.yaml` is no longer used by the active lane lifecycle. Do not recreate
  or maintain it on upgrade. Use `workspace.yaml > software_delivery.lanes.definitions` as the sole
  source of truth, and use `pnpm lane:suggest` only to generate workspace lane-definition snippets.
</Aside>

<Aside type="note" title="Defaults vs overrides">
  The installed CLI ships the default `wu:brief` composition. Consumers do not need a local
  `.lumenflow/templates/manifest.yaml` just to get lane-aware UI guidance, structured-content
  guidance, or required-verification rendering.
</Aside>

## New Installs

Fresh installs already get the current defaults. Run `npx lumenflow` or
`pnpm lumenflow`, then create project-local templates only when you want custom prompt
sections or client-specific overrides.

## File Ownership Model

| File                           | Owner                     | On Upgrade                                                      | User Overrides           |
| ------------------------------ | ------------------------- | --------------------------------------------------------------- | ------------------------ |
| `LUMENFLOW.md`                 | LumenFlow (fully managed) | Force-synced from template                                      | Use `LUMENFLOW.local.md` |
| `.lumenflow/constraints.md`    | LumenFlow (fully managed) | Force-synced from template                                      | —                        |
| `AGENTS.md`                    | Shared (merge-block)      | Content between `LUMENFLOW:START`/`END` updated; rest preserved | Edit outside markers     |
| `CLAUDE.md`                    | Shared (merge-block)      | Content between markers updated; rest preserved                 | Edit outside markers     |
| `.cursor/rules/lumenflow.md`   | Shared (merge-block)      | Content between markers updated; rest preserved                 | Edit outside markers     |
| `.windsurf/rules/lumenflow.md` | Shared (merge-block)      | Content between markers updated; rest preserved                 | Edit outside markers     |
| `LUMENFLOW.local.md`           | User (never touched)      | Skipped entirely                                                | Full ownership           |
| `.lumenflow/templates/`        | User (never touched)      | Skipped entirely                                                | Full ownership           |

<Aside type="note" title="LUMENFLOW.local.md">
  Create `LUMENFLOW.local.md` for project-specific workflow additions. Agents read it after
  `LUMENFLOW.md` when present. It is never overwritten by `init` or `upgrade`.
</Aside>

## What `docs:sync` Refreshes

`docs:sync` is now run automatically by `lumenflow:upgrade`. You can also run it standalone:

- **Managed docs** (`LUMENFLOW.md`, `.lumenflow/constraints.md`) — always force-synced from templates
- **Bootstrap docs** (`AGENTS.md`, vendor files) — merge-block update only (user content preserved)
- Onboarding documentation and vendor assets (skills, hooks)

<Aside type="caution" title="Standalone docs:sync">
  Running `pnpm docs:sync` standalone is only needed when you want to refresh docs without upgrading
  packages (e.g., after manually editing templates). For normal upgrades, `pnpm lumenflow:upgrade
  --latest` handles everything.
</Aside>

## Config Contract

- Runtime reads: `workspace.yaml > software_delivery`
- Config write command:

```bash
pnpm config:set --key software_delivery.gates.minCoverage --value 90
```

- Config read command:

```bash
pnpm config:get --key software_delivery.methodology.testing
```

<Aside type="caution" title="Canonical command">
  Use `npx lumenflow` for setup and `pnpm lumenflow:upgrade` for upgrades.
</Aside>

## Troubleshooting

### `workspace.yaml` missing

```bash
pnpm workspace-init --yes
```

### Older scripts call split onboarding commands

Replace them with:

```bash
npx lumenflow
```

### `wu:brief` still shows older project wording

Project-local overrides take precedence over the built-in defaults. Review:

- `.lumenflow/templates/manifest.yaml`
- `.lumenflow/templates/spawn-prompt/*.md`
- `.lumenflow/templates.<client>/spawn-prompt/*.md`
